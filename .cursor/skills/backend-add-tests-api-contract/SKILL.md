# Backend Add Tests API Contract

Писать API-тесты так, чтобы они проверяли поведение ручек как публичный контракт, а не детали реализации.

## Workflow

1. Зафиксировать контракт endpoint.
- Определить путь, HTTP-метод, версию API (`api/v1/...` по умолчанию), success/error статусы.
- Определить expected envelope, обязательные поля и `error code`.

2. Собрать минимальный тест-набор.
- Success-кейс (`200/201/204` по контракту).
- Auth-кейс: `401` без JWT-токена или с невалидным.
- Validation/business error кейсы (`400/404/409` и т.д.).

3. Проверить контракт ошибок.
- Проверить не только статус, но и машинный `code` и формат error-envelope.
- Убедиться, что runtime-ошибки соответствуют OpenAPI-декларации.

4. Проверить побочные эффекты.
- Для mutating endpoint проверить состояние БД/кэша/очередей/внешних вызовов.
- Изолировать внешние интеграции моками.

5. Проверить версионирование.
- Тестировать endpoint в рамках активной версии (`v1`, если пользователь не указал другую).
- Не добавлять тесты под `v2+` без явного решения пользователя о смене версии.

6. Запустить проверки.
- `uv run pytest` — целевые тесты, затем полный набор.
- `uv run ruff check .` + `uv run mypy .`.

## Test Matrix

- `happy path`
- `unauthorized/forbidden`
- `validation error`
- `not found`
- `conflict` (если применимо)
- `side effects`
- `OpenAPI parity` (status + error-code)

## Definition Of Done

- Тесты покрывают основной и негативные сценарии контракта.
- Runtime поведение совпадает с OpenAPI.
- Побочные эффекты проверены.
- Тесты и проверки проекта проходят.
