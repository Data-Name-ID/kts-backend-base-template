# Backend Auth Guard Context

Реализовывать безопасность и контекст доступа последовательно по всему API.

## Workflow

1. Зафиксировать модель доступа.
- Определить субъект (`user`), ресурс, действие и policy.
- Определить границы: публичные ручки (`exclude_from_auth=True`) и авторизованные ручки.

2. Определить точку применения проверки.
- Аутентификация глобальная через `JWTAuth[UserAuth]` в `main.py`.
- Публичные ручки явно помечать `exclude_from_auth=True` в декораторе endpoint.
- Авторизацию на уровне ресурсов выполнять в handler или manager, не дублировать в каждом endpoint.

3. Настроить request-context.
- Текущий пользователь доступен как `request.user` типа `UserAuth(id: int)`.
- Получать идентификатор: `user_id = request.user.id`.
- `Store` передаётся через DI (`store: Store` в сигнатуре), не инстанцировать вручную.

4. Зафиксировать поведение ошибок доступа.
- `401` для отсутствующей/невалидной аутентификации.
- `403` для недостаточных прав при валидной аутентификации.
- Ошибки отдавать в едином error-envelope и декларировать в OpenAPI.

5. Проверить версионирование API.
- Применять правила доступа в рамках актуальной версии (`api/v1/...` по умолчанию).
- Не переносить доступ в новую версию API (`v2+`) без явного решения пользователя.

6. Добавить тесты безопасности.
- Кейсы: anonymous (`401`), authenticated-has-access (`200/201/204`).
- Запуск: `uv run pytest`.

## Anti-Patterns

- Смешивать auth и бизнес-валидацию в одном месте без явных границ.
- Возвращать `403` там, где должна быть `401`, и наоборот.
- Размазывать policy-логику по контроллерам.
- Дублировать парсинг контекста в каждом endpoint.

## Definition Of Done

- Модель доступа реализована консистентно через guard/middleware/context.
- `401/403` семантика стабильна и документирована.
- OpenAPI отражает ошибки доступа.
- Тесты безопасности покрывают ключевые сценарии.
